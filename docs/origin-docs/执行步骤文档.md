SmartSleep Alarm v0.2 文档。v0.2 的定位已经很清楚：它不是“做一个完整睡眠 App”的需求文档，而是“先验证 Apple 生态智能闹铃链路是否成立”的 Spike-first PRD。文档明确把核心目标放在 iPhone 创建闹铃、Watch 当晚/当次启用确认、Watch runtime session、motion-first 监测、iPhone 兜底、日志回放和内部 dogfood 上；自动静音、防再睡、心率增强也被降级为 P0-Experiment，而不是第一版必须稳定承诺的能力。

核心结论：下一步不要直接让 Codex “帮我实现 MVP”。正确做法是把 v0.2 拆成工程合同、Spike 任务、状态机、日志模型、测试矩阵和小 PR。Codex 负责高效生成代码、测试、脚手架和文档；你负责收紧范围、验收 Apple 真机行为、决定哪些能力进入 MVP。

## 1. 先把 v0.2 变成工程资产，而不是继续写 PRD

第一步是在代码仓库里建立一个“单一事实来源”。建议结构如下：

```text
SmartSleepAlarm/
  docs/
    prd/
      SmartSleep_Alarm_v0.2.md
    adr/
      ADR-0001-v02-scope-and-non-goals.md

    spikes/
      Spike-A-watch-runtime-session.md
      Spike-B-alarm-channel-reliability.md
      Spike-C-motion-awake-candidate.md

      Spike-E-gesture-snooze.md
      Spike-F-privacy-review.md
    qa/
      device-test-matrix.md
      dogfood-runbook.md

  Packages/
    SmartSleepCore/
    SmartSleepDiagnostics/

```

v0.2 文档里已经给出 iOS App、watchOS App、WatchConnectivity 的职责边界：iOS 负责闹铃 CRUD、权限引导、Watch 同步、iPhone 兜底、日志展示和导出；Watch 负责启用确认、runtime session、motion/HR 采集、响铃、自动静音、手势贪睡、防再睡状态机和本地日志；WatchConnectivity 负责两端配置、状态、日志和 outcome 同步。这个模块划分可以直接变成 Xcode workspace 和 Swift Package 的边界。

AGENTS.md 要先写。它是你给 Codex 的项目宪法。建议第一版内容类似这样：

```text
You are working on SmartSleep Alarm v0.2.

Rules:
- Implement v0.2 only. Do not add sleep reports, cloud sync, ML personalization, audio recording, Android, or medical features.
- Reliability beats intelligence. If uncertain, keep ringing.
- Smart Mode 只有在 Watch 完成启用确认并成功预约 runtime session 后，才能显示为 Ready。
- Motion is the primary signal. HealthKit heart rate is optional enhancement only and must never trigger auto-silence alone.
- Every state transition, alarm channel event, sensor freshness event, gesture event, and outcome must be logged.
- Auto silence and re-sleep detection are P0-Experiment behind feature flags.
- No workout-session workaround to force high-frequency heart rate.
- Keep PRs small. Add tests for pure Swift logic. Do not change unrelated files.
- For Apple API code, add a manual device-test note if behavior cannot be verified in simulator.
```

这一步非常关键。没有 AGENTS.md，Codex 很容易把项目写成“普通闹钟 + 睡眠报告 + AI 算法”的泛化产品；有了 AGENTS.md，Codex 每次改代码都会被你的 v0.2 边界约束住。

## 2. 技术路线：先做“可测试核心”，再碰 Apple 难点

建议把开发分成四层。

第一层是纯 Swift 核心层，不依赖 iOS/watchOS API。这里包括 `Alarm`、`AlarmRun`、`DeviceContext`、`StateTransitionLog`、`SensorFreshness`、`GestureEvent`、`OutcomeLabel` 等数据模型，以及主状态机、自动静音评分、防再睡评分、手势识别器。v0.2 已经给出了这些模型和状态机，可以直接转成 Swift `Codable` 类型和单元测试。

第二层是诊断与日志层。这个项目早期最大的资产不是 UI，而是真机日志。建议先做 append-only JSONL 本地日志，后面再加 SwiftData/SQLite 索引。每个事件都应该能回答五个问题：什么时候发生、在哪个设备发生、处于哪个状态、使用了哪个通道、为什么做这个判断。

第三层是 Apple 集成层。这里才接入 AlarmKit、WatchConnectivity、WKExtendedRuntimeSession、CoreMotion、HealthKit。Apple 的 AlarmKit 用于管理带可定制 schedule 和 UI 的闹铃，适合作为 iPhone 侧兜底；WatchConnectivity 是 iOS app 与配对 watchOS app 之间传输数据的官方框架；watchOS extended runtime session 可以让 Watch App 在用户停止交互后继续运行，用于处理数据、声音或触觉等后台场景；Core Motion 的 `CMMotionManager` 用于读取设备传感器检测到的运动数据。([Apple Developer][1])

第四层是产品 UI。早期 UI 只服务于链路验证：创建闹铃、Watch 启用确认、显示 Ready/Fallback/Failed、响铃页、手动停止/贪睡、日志导出、反馈标注。不要先打磨完整 onboarding、视觉品牌、睡眠分析页或订阅页。

推荐顺序是：

```text
SmartSleepCore
  -> StateMachine
  -> EventLog
  -> Replay Harness
  -> iOS Alarm CRUD skeleton
  -> Watch enable-confirmation skeleton
  -> WatchConnectivity sync
  -> Runtime session Spike
  -> Alarm channel Spike
  -> Motion collection Spike
  -> Gesture Spike
  -> HR freshness Spike
  -> Vertical Slice MVP
  -> Dogfood
```

## 3. 用 v0.2 拆任务：不要按功能拆，要按风险拆

v0.2 的工程判断是“难点不在 UI，也不在第一版算法有多复杂，而在 Apple 生态链路是否足够可靠”。所以任务优先级应该按风险排序，而不是按用户界面排序。

第一优先级是 Spike A 和 Spike B：Watch 端启用确认与 runtime session、闹铃通道可靠性。文档已经列出要测试的场景，包括单个/多个/重复闹铃、距离闹铃不同时间、Watch 锁屏/重启/force quit/低电量/断连/未佩戴/充电，以及 Silent Mode、Sleep Focus、Do Not Disturb、锁屏、飞行模式、App 后台/终止等。Spike 的退出标准不是“代码写完”，而是得到支持/降级矩阵，并且所有失败原因都能被日志捕获。

第二优先级是“普通响铃闭环”，暂时不要启用自动静音。目标是完成一个纵向切片：

```text
iPhone 创建闹铃
  -> 同步到 Watch
  -> Watch 显示 等待 Watch 确认
  -> 用户在 Watch 启用确认
  -> Watch 预约 runtime session
  -> iPhone 创建兜底通道
  -> 到点 Watch 响铃/震动
  -> iPhone 兜底也按策略触发或记录
  -> 用户手动停止
  -> 两端记录 AlarmRun / AlarmChannelLog / StateTransitionLog
  -> iPhone 导出 JSON
```

第三优先级才是 motion-first 自动静音、手势贪睡、防再睡风险。自动静音在 v0.2 中是 P0-Experiment，只能在 `RINGING` 状态触发，先进入 `AWAKE_CANDIDATE`，经过 3 秒候选窗口和 10–20 秒确认窗口后，高置信度才静音；心率只能加分，不能单独触发；motion stale 时不得自动静音。

第四优先级是 HealthKit 心率新鲜度。HealthKit 应作为增强而不是依赖。Apple 的 HealthKit 是在用户授权后让 App 读写健康和健身数据的框架；这和 v0.2 的“心率只作机会性增强、不能作为秒级实时强依赖”的策略一致。([Apple Developer][2]) 

## 4. Codex / vibe coding 工作流

你有 MacBook Air M4、Xcode、ChatGPT Pro 的 Codex。建议用“本地 Codex + Xcode 真机验证 + Git 小 PR”的组合。OpenAI 的 Codex app 支持 macOS，ChatGPT Pro 计划包含 Codex；Codex app 可以选择项目文件夹，在本机 Local 模式工作，并支持 Git/worktree、终端命令、review diffs 等流程。([OpenAI开发者][3])

vibe coding 在这个项目里可以用，但必须加工程护栏。正确节奏是：

```text
人写约束和验收
  -> Codex 生成最小实现
  -> xcodebuild / 单元测试
  -> 人审 diff
  -> 真机验证
  -> 记录 Spike 结果
  -> 再进入下一步
```

不要让 Codex 一次性写“完整 SmartSleep Alarm App”。每次任务应该控制在一个小 PR，例如：

“实现 `AlarmStateMachine`，只包含状态枚举、事件枚举、transition 函数和单元测试。不要写 UI，不要接 Apple API。”

“根据 v0.2 的数据模型实现 `AlarmRunEventStore`，使用 JSONL 本地追加日志，提供 export 方法。每个 event 必须带 schemaVersion、deviceRole、timestamp、runId。”

“实现 WatchConnectivity 的 payload 类型和 mock transport，先不要接真实 `WCSession`，给出 iOS->Watch alarm config 和 Watch->iOS enable-confirmation result 的测试。”

“实现 `AwakeScorer`，输入 `SensorSummary` 和 `SensorFreshness`，输出 score、confidence、reasonCodes。心率不得单独触发自动静音。补充测试：single spike 不静音、HR-only 不静音、motion stale 不静音。”

“实现 `GestureSnoozeDetector`，只在 `RINGING` / `RE_RINGING` 有效，支持 cooldown、1.5 秒窗口、两次方向一致旋转。补充 reject/accept 测试。”

每次给 Codex 的 prompt 都要包含四个要素：目标文件、引用的 v0.2 章节、明确不做什么、验收方式。vibe coding 的优势是快，但这个项目不能靠“看起来能跑”来判断成功，因为闹铃产品最怕的是边缘状态：没响、误静音、误贪睡、日志缺失。

## 5. 第一批开发任务清单

建议你从下面 12 个任务开始，顺序不要乱。

1. 建 Git 仓库和 Xcode workspace。包含 iOS App target、watchOS App target、`SmartSleepCore` Swift Package、`SmartSleepDiagnostics` Swift Package。

2. 把 v0.2 放入 `/docs/prd/`，写 `AGENTS.md`，写 ADR-0001 到 ADR-0004。ADR 不需要长，但要把“Watch 必须完成启用确认”“motion-first”“iPhone 兜底不可省”“不做睡眠报告/医疗”钉死。

3. 用 Codex 生成 `SmartSleepCore` 的数据模型：`Alarm`、`AlarmRun`、`DeviceContext`、`RuntimeSessionLog`、`AlarmChannelLog`、`SensorFreshness`、`SensorSummary`、`StateTransitionLog`、`GestureEvent`、`OutcomeLabel`。

4. 用 Codex 实现纯 Swift 状态机。状态必须覆盖 `IDLE`、`CREATED_ON_PHONE`、`NEEDS_WATCH_ARMING`、`ARMED_ON_WATCH`、`SESSION_SCHEDULED`、`PRE_MONITORING`、`RINGING`、`AWAKE_CANDIDATE`、`SILENCED_MONITORING`、`RE_SLEEP_RISK_DETECTED`、`RE_RINGING`、`SNOOZED`、`COMPLETED`、`FALLBACK_PHONE_ALARM`、`RINGING_NO_SMART`。v0.2 已经给出主路径、贪睡路径、防再睡路径和异常路径，可以直接转测试。

5. 实现日志系统。先用 JSONL，后续再决定 SwiftData。最低要求：每次状态转换、闹铃通道触发、传感器新鲜度变化、手势 accept/reject、用户反馈都能落盘。

6. 做 iOS 闹铃 CRUD 骨架。只做时间、重复、标签、Smart Mode 开关、贪睡间隔、兜底偏好。列表状态必须区分 `Smart Off`、`需在 Watch 上启用`、`Ready`、`Fallback Only`、`Failed`，不能在 Watch 未完成启用确认时显示 Ready。

7. 做 Watch 端启用确认页。Watch 收到配置后，显示下一次闹铃、权限/佩戴/电量/兜底状态，用户点击“今晚启用”，然后记录启用确认尝试。

8. 接 WatchConnectivity。先传最小 payload：alarm config、enable-confirmation result、session result、run log summary。断连时两端都不能阻塞执行。

9. 做 Spike A：Watch runtime session。重点验证预约、启动延迟、是否能按时进入预监测、失败原因、force quit/锁屏/低电量/断连等场景。

10. 做 Spike B：闹铃通道。先验证 Watch haptic/audio、iPhone AlarmKit 或降级通知、Local Notification、前台音频、人工兜底提示的表现。v0.2 要求 iOS 26+ 且授权时优先用 AlarmKit，不满足时降级，并且任何兜底通道触发都要写入 `AlarmChannelLog`。

11. 做“无智能判断”的 Vertical Slice。只要能稳定创建、启用确认、响铃、手动停止、导出日志，就可以进入内部 dogfood 的第一阶段。

12. 再接自动静音、手势贪睡、防再睡。全部放在 feature flag 后面，默认保守，宁可不触发，也不能误触发。


## 7. Spike-first 的具体执行方式

Spike A 的输出不是代码本身，而是这个表：

```text
场景 | 预约是否成功 | 实际启动时间 | 启动延迟 | 是否到达响铃 | 失败原因 | 是否触发 iPhone 兜底 | 日志是否完整 | 结论
```

Spike B 的输出是通道可靠性矩阵：

```text
场景 | watchRuntimeHapticAudio | iOSAlarmKit | iOSLocalNotification | foregroundAudio | manualFallbackPrompt | 推荐策略
```

Spike C/E 的输出是 motion/gesture 样本库：

```text
动作 | 是否应该触发 | 实际 score | reasonCodes | 是否误判 | 原始窗口 JSON
```

Spike D 的输出是心率新鲜度分布：

```text
场景 | hrAgeSec p50 | p90 | p95 | baseline confidence | 是否参与评分
```

Spike F 的输出是审核和隐私材料：

```text
权限文案 | 非医疗声明 | 数据用途 | 本地存储 | 删除入口 | TestFlight 描述 | App Store 元数据风险
```

这类表格比“功能完成百分比”更重要，因为 v0.2 真正要回答的是：Apple 链路能不能支撑这个产品，而不是 UI 是否已经像正式产品。

## 8. MVP 不应做的事情

不要做睡眠报告页。不要做睡眠阶段识别。不要做录音、打鼾检测、助眠内容、云同步、跨用户训练、订阅墙、复杂任务闹钟、Android。文档明确把这些排除在 v0.2 之外。

不要伪造 Workout session 来提高心率频率。v0.2 已经明确 HealthKit 心率不能作为实时强依赖，且不应为了提高心率频率伪造 workout session。

不要过早打磨算法。第一版算法应该是可解释启发式，不是 ML。你现在最需要的是日志、回放、样本和误判标注。

不要让 iPhone 兜底变成“以后再说”。闹铃产品的第一原则是不能漏叫。v0.2 的 MVP 原则已经明确：可靠性优先于智能感，不确定就继续响，Watch 独立执行优先，iPhone 兜底必须明确，每次自动判断都必须可解释。

## 9. 内部 dogfood 入口标准

进入内部 dogfood 前，至少满足：

```text
1. iPhone 能创建/删除/禁用闹铃。
2. Watch 能接收配置并完成启用确认。
3. Watch 未完成启用确认时，iPhone 不显示 Smart Mode Ready。
4. Watch session 结果可记录。
5. iPhone 兜底通道已创建并可记录触发结果。
6. 到点至少一个通道会响。
7. 手动停止和手动贪睡可用。
8. 每次 AlarmRun 可导出 JSON。
9. HealthKit 未授权时仍可 motion-only。
10. Motion 不可用时自动静音和手势贪睡禁用。
11. 自动静音、防再睡、手势贪睡均有 feature flag。
12. 失败能归因到 session、传感器、通道、权限、系统状态或用户操作。
```

v0.2 的工程验收目标很具体：内部 100 次 dogfood 闹铃中 `missedAlarm` 必须为 0；自动静音误判目标低于 3–5%；误重响目标低于 10–15%；motion stale 或 Watch 未佩戴时 false silence 必须为 0；Watch 未完成启用确认不允许显示 Smart Mode Ready。

## 10. 下一步最推荐的执行顺序

直接按这个顺序启动：

第一，创建 repo、workspace、Swift Package、AGENTS.md，把 v0.2 放进 `/docs/prd/`。

第二，让 Codex 先做 `SmartSleepCore`：数据模型、状态机、日志事件、单元测试。不要让它碰 UI。

第三，让 Codex 做 iOS/Watch 的空壳 UI：iOS 闹铃列表、创建页、状态标签；Watch 启用确认页、响铃页、日志同步占位。

第四，接 WatchConnectivity 的真实同步，但只传最小 payload。

第五，做 Spike A/B。没有完成 Watch runtime 和 alarm channel 可靠性矩阵前，不进入自动静音开发。

第六，完成“无智能判断”的纵向切片并 dogfood。先证明不会漏叫。

第七，接 motion 采样、手势检测、自动静音、防再睡。全部通过 feature flag 控制，所有判断必须写日志。

第八，用 dogfood 日志回放调阈值。v0.3 再做稳定性和阈值调优，而不是在 v0.2 里追求“看起来很聪明”。

最重要的工程原则是：把 v0.2 当作“可验证假设清单”。Codex 加速写代码，但不要让它替你验证 Apple Watch 在真实夜间场景里的系统行为。这个项目的成败点是可靠链路和可观测性，不是第一版 UI 或算法复杂度。
